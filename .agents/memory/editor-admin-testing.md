---
name: Testing the Visual Editor / creating the first admin
description: How to create the first CMS user and exercise the /editor save flow when verifying this Payload app.
---

- **First admin user:** `POST /api/users/first-register` (plain `POST /api/users` is blocked — the `Users` collection only overrides `read` access, so `create` falls back to the default that requires an already-logged-in user). The editor API authenticates either the `payload-token` cookie or an `Authorization: JWT <token>` header.

- **Dev and production use SEPARATE databases on Replit.** Admin users / seed data created in development do NOT exist in production. Production data is created by code that runs inside the deployed app, via `payload.config.ts` `onInit`, which idempotently: (a) creates a static admin (`src/lib/bootstrapAdmin.ts`, default `zaidjalal@gccappadmin.com`, override with `ADMIN_EMAIL`) — only when missing, so a dashboard password change survives redeploys; and (b) auto-imports the built-in bilingual content into the CMS collections **once per database** (`autoSeedOnInit` in `src/lib/seedContent.ts`, guarded by an `app_seed_state` marker row + non-blocking `pg_try_advisory_lock`). **Why once-only:** re-running on every cold start would resurrect content the user intentionally deleted, and concurrent Autoscale cold-starts could otherwise double-seed (slugs are NOT unique-indexed). The manual "Import website content" button still calls the pure `importWebsiteContent` for on-demand re-import. **Note:** auto-seed fills collections only — it does NOT seed the `home-page` global, so the logo-marquee items must be added live in `/editor`. Changing prod content/credentials requires a republish (production SQL is read-only from the agent).

- **server-only data layers can't be unit-tested standalone:** `src/lib/cms.ts` and `src/lib/homePage.ts` begin with `import "server-only"`, which throws in a plain node / `payload run` / tsx script (no `react-server` import condition). Exercise `saveHomeSections`/`loadHomeSections` through the HTTP editor API (`GET`/`POST /api/editor/home`) instead, which runs in the Next.js server context.

- **First-save migration is identical-by-design:** the `/editor` first save migrates the inline home defaults into the `home-page` global; the visible page text is byte-for-byte unchanged before/after in both locales (only internal RSC chunk ids / section UUIDs change).

- **Object Storage vs Postgres media:** media uploads use the Replit Object Storage bucket only when `OBJECT_STORAGE_ENABLED` is set AND a bucket exists in `.replit`; otherwise `src/lib/postgresStorage.ts` stores bytes in the `media_blobs` table (served via `/api/media/file/<filename>`). Never set `OBJECT_STORAGE_ENABLED=true` before a bucket exists — `new Client()` throws "A bucket name is needed".

- **`payload generate:importmap` CLI is broken in this env:** it crashes with undici `CacheStorage` "Illegal constructor" (config loaded via tsx). The committed `src/app/(payload)/admin/importMap.js` already registers the only custom admin component (`Welcome`, which itself imports `SeedButton` as a normal React import), and the Next dev server runs fine. Don't regenerate or treat the "run generate:importmap if missing-component error" note as a real problem unless the admin UI actually shows that error.

- **A newly added default home section does NOT appear on an already-saved home page.** Once `/editor` has been saved, `loadHomeSections` returns the stored `home-page` global (`source: "global"`) and ignores `buildDefaultSections`. So adding a new block to the defaults (e.g. the "Client logos" section after the hero) only auto-appears on a site whose global is still empty (`source: "default"`). On a site that already saved its home page, the new section must be added by hand in the editor ("Add section"). **Why:** the saved global is the source of truth; defaults are only the pre-first-save fallback. **How to apply:** when verifying/adding a new home block, GET `/api/editor/home` first — if `source==="global"` you must splice the block into the returned `sections` (insert after the hero) and POST it back; localized fields like `eyebrow` need a POST per locale (`ar` and `en`) because the non-first-save path skips the other-locale backfill.

- **Media upload via REST** (`POST /api/media`, multipart): put document fields (e.g. required `alt`) in a `_payload` JSON part — `fd.append("_payload", JSON.stringify({alt}))`. A plain `fd.append("alt", ...)` is ignored and fails validation. Uploaded media URLs are `/api/media/file/<name>.webp?prefix=media` (the `?prefix=media` is required).
